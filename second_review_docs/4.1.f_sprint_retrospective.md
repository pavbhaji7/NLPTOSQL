# 4.1.f Sprint Retrospective

*This document outlines a standard agile retrospective following the completion of the main implementation phase of G-SQL.*

## Retrospective Details
**Sprint Name:** Initial G-SQL Core & UI Integration
**Duration:** 2 Weeks

---

## What Went Well? (Successes)
1. **Rule-Guided Parsing Integration:** Utilizing `spaCy` to parse complex string arrays via lemmatization and tagging went much smoother than creating an original parser logic tree.
2. **FastAPI Decoupling:** Separating the internal translation Python code from the React frontend via a REST API allowed the user interface to be incredibly responsive without lagging the visual pipeline components.
3. **Graph-Based Resolution Implementation:** Building adjacency lists from the Foreign Key definitions proved a highly effective heuristic for the system's ability to self-author `JOIN ON` SQL commands.

---

## What Didn't Go Well? (Challenges)
1. **Handling Value Ambiguity:** Distinguishing whether a token like "Titanic" was a "Meta" (table name like a table named Titanic, which is wrong) or a "Value" required more static dictionary lookup definitions than originally planned. A larger sample schema highlights limitations in the simple classification approach.
2. **CSS Layout for Visualizer:** Constructing dynamic, color-coded arrays using vanilla CSS flex properties caused issues handling responsive query length wraps, which delayed UI presentation polish.
3. **Implicit Conditions:** NLP inputs asking for "best" or "top" (implicit `ORDER BY` + `LIMIT`) require extremely exhaustive heuristic lists.

---

## Action Items for Next Phase
- **Expand the Schema Model:** Move past the IMDB sample database to test G-SQL string translation with varied enterprise schemas (finance, HR, etc.).
- **Develop "Context Memory":** If the user asks a follow-up question ("What was its budget?"), the system currently loses the `Table` state. Building context persistence within the FastAPI routing logic is essential for Chat-like interaction.
- **Implement Robust Value-Type Verification:** Prevent the system from matching string Values to structural `INT` columns. The `DatabaseSchema` mapping presently does not enforce strict data typing on matched inputs.
