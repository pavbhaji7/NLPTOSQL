# Project Creation Guide:(Natural Language to SQL)

This document outlines how the project was conceptualized, structured, and implemented. It provides a comprehensive overview of the architecture, the rule-guided NLP pipeline, and the frontend visualization, making it an ideal reference for generating textual answers about the project.

---

## 1. Project Inspiration and Goal
The goal of this project was to create a robust, schema-aware system that translates Natural Language Queries (NLQ) directly into SQL queries. 

Unlike modern "black-box" Large Language Model (LLM) approaches that generate SQL unpredictably, this project was built on a **transparent, rule-guided pipeline.** The intention was to allow users to ask questions like *"List actors who played in Avatar"* and see exactly how the system broke down that sentence logically to generate a functional SQL query.

---

## 2. Core Architecture
The system was divided into two main components to establish a modern decoupled architecture:
1. **The Core Python Engine (Backend):** Handles Natural Language Processing (NLP), Semantic Tagging, Schema Linking, Join Inference, and SQL Generation.
2. **The React Visualization App (Frontend):** Provides a modern, interactive User Interface (UI) where users can type queries, view the database schema, and see visual "chips" representing how the NLP pipeline processed their question.

---

## 3. How the Backend Was Built (The G-SQL Engine)
The backend was developed using **Python** and was structured into distinct modular stages within a `gsql` library. We used **FastAPI** to expose these internal modules as a REST API.

### Stage 1: Natural Language Processing (`nlp.py`)
- **Technology Used:** We used `spaCy` (specifically the `en_core_web_sm` model).
- **Process:** The system takes the raw user text and breaks it down into "Tokens." It analyzes each word's part-of-speech (Noun, Verb), its root lemma (e.g., "released" becomes "release"), and its dependency in the sentence structure.

### Stage 2: Semantic Tagging (`tagger.py`)
- **Process:** We created a rule-based algorithm to assign specific **G-SQL Tags** to the NLP tokens. We built dictionaries to categorize words into:
  - **Meta:** Database tables or columns (e.g., "Movie", "budget").
  - **Value:** Specific data points the user is searching for (e.g., "Avatar", "2000").
  - **Cond:** Conditions and operators (e.g., "greater", "after", ">").
  - **AGG:** Aggregations (e.g., "sum", "average").

### Stage 3: Schema Linking & Graph-Based Joins (`linker.py`)
- **Process:** This is the most complex logical component. We built a system to map the "Meta" and "Value" tags directly to our predefined mathematical `DatabaseSchema` (like Movies, Actors, Directors). 
- **Graph-Based Inference:** To handle complex questions requiring data from multiple tables (e.g., "Find the director of Titanic"), we implemented a **Breadth-First Search (BFS)** algorithm over an adjacency graph of Primary Key/Foreign Key relationships. This allows the system to automatically infer which `JOIN` statements are required to connect distant database tables.

### Stage 4: SQL Generation (`generator.py`)
- **Process:** A structured `SQLGenerator` class takes the fully linked and mapped structure (SELECT targets, FROM sets, WHERE conditions, and inferred JOINs) and formats it into a final, syntactically correct SQL string.

### Stage 5: The API Layer (`server.py`)
- **Technology Used:** `FastAPI` and `Uvicorn`.
- **Process:** We built a REST endpoint (`/api/translate`) that runs the 4 stages above in sequence and returns a structured JSON payload to the frontend. This payload includes not just the final SQL string, but all the intermediate states (the tagged tokens, the mapped schema elements) so the frontend could explain the "why."

---

## 4. How the Frontend Was Built (The Visualization UI)
The frontend was developed to make the complex backend logic transparent.

- **Technology Stack:** We initialized a modern Single Page Application (SPA) using **Vite + React.js**, styling it entirely with **Vanilla CSS** within `index.css` to build an independent, sleek design system without relying on heavy frameworks like Bootstrap or Tailwind.
- **Component Architecture:** We broke the UI down into modular React components:
  - `SchemaViewer.jsx`: A graphical representation of the database tables and their foreign key relationships, placed on the left side of the screen.
  - `QueryInput.jsx`: A stylized search bar in the center for user input.
  - `PipelineVisualizer.jsx`: The core visual element. It consumes the intermediate token data returned by FastAPI and renders color-coded "chips" (e.g., Blue for Meta, Green for Values). This is how users see *how* their sentence was analyzed.
  - `SQLResult.jsx`: A syntax-highlighted block that displays the final output SQL string generated by the Python engine.

---

## 5. Technical Highlights and "Improvisation"
- **Avoidance of "Black-Box" AI:** While LLMs are popular for NL-to-SQL, they are prone to hallucinations and are difficult to debug. By building a deterministically rule-guided pipeline (tagging -> linking -> graph-based joins), this project guarantees that the generated SQL strictly adheres to the provided schema.
- **Seamless Local Execution:** The project avoids heavy cloud dependencies. By relying on lightweight local NLP models (`spaCy`) and local rule sets, the entire inference process happens rapidly on the user's local machine via the Python server.
- **Automated Relationship Mapping:** The implementation of the BFS algorithm in `linker.py` to automatically trace foreign-key paths and build `JOIN` statements represents a significant engineering achievement tailored specifically for this schema-aware approach.
